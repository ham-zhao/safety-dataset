# ML 项目标准方法论

> 从一个完整 ML 项目的全流程实践中提炼的可复用方法论。覆盖项目五阶段，每条原则附带事前/事中/事后检查和真实案例。
> 适用于：预训练数据清洗、模型训练、评估对比等 ML 项目。

---

## 使用说明

### 元原则：中性化

本文档所有规则的表述与具体项目、工具、框架解耦。使用通用术语而非特定技术名词。项目特定案例仅作为括号内示例出现。目的：确保方法论可直接复用于任何 ML 项目，无需修改。

### 三阶段框架

每条方法论均包含三阶段检查：
- **事前（Pre）**：开始前确认什么条件 → 防止方向错误
- **事中（During）**：执行中检查什么状态 → 防止过程偏离
- **事后（Post）**：完成后验收什么标准 → 防止遗漏和回归

### 文档层级

| 层级 | 文件 | 内容 | 生效范围 |
|------|------|------|---------|
| 全局规则 | `~/.claude/CLAUDE.md` | Agent 执行行为规则 | 所有项目自动生效 |
| 项目规则 | `项目/.claude/MEMORY.md` | 项目特有状态和决策 | 仅当前项目 |
| 方法论 | `docs/STANDARD_METHODOLOGY.md`（本文件） | 完整方法论知识库 | 按需引用 |
| 精简提示 | `docs/METHODOLOGY_COMPACT.md` | <500 字 prompt 前缀 | 写入 CLAUDE.md 激活 |

标注 **[→ CLAUDE.md]** 的规则需同步写入 Agent 行为规则文件，否则 Agent 不会自动执行。

## 一、理解与设计阶段（Understand & Design）

### 1.1 五要素定义

**原则：每个 Pipeline 阶段 / 每个分析单元开始前，必须先明确五要素。**

```
① 输入：什么数据，什么格式，从哪来，多少条
② 输出：产出什么，给谁用，什么格式，交付标准
③ 过程：经过哪些步骤，每步做什么
④ 目标：这个环节要达成什么效果
⑤ 度量：用什么指标衡量，基准值（论文/业界）是多少，预期效果是多少
```

**三阶段检查：**

| 阶段 | 检查内容 |
|------|---------|
| 事前 | 五要素是否全部填写？度量标准是否有基准值来源？ |
| 事中 | 实际执行是否偏离了预定义的过程步骤？ |
| 事后 | 输出是否满足预定义的交付标准？度量指标是否达到预期？ |

> **案例**：某项目的数据清洗模块，事先未定义"度量标准"。执行完后无法判断清洗效果好不好——因为从未定义过"好"的标准是什么。补充定义后，能立即判断"过滤率 58% 在论文参考范围 40-60% 内，符合预期"。

---

### 1.2 MECE 框架完整性检查

**原则：项目拆解为子模块时，确保相互独立（ME）且完全穷尽（CE）。**

> MECE = Mutually Exclusive, Collectively Exhaustive（相互独立，完全穷尽）。来源：McKinsey, Barbara Minto《金字塔原理》。

**检查方法：**
- **ME 检查**："有没有两个模块做同一件事？" → 职责重叠则合并
- **CE 检查**："有没有关键环节没人负责？" → 遗漏则补充

**三阶段检查：**

| 阶段 | 检查内容 |
|------|---------|
| 事前 | 画出完整数据流图 → 每个环节标注负责的模块 → ME + CE 自检 |
| 事中 | 新增模块时检查：是否与已有模块职责重叠？ |
| 事后 | 端到端走查：数据从输入到最终输出，有没有断点？ |

> **案例**：某项目两个分析模块都包含"数据去重"逻辑（违反 ME），而"数据增强"环节无人负责（违反 CE）。重构后每个模块职责唯一，增补数据增强模块。

---

### 1.3 假设合理性检查

**原则：每个论证环节的假设必须显式声明，偏离业界标准的假设必须说明理由。**

```
如果假设和论文/业界标准有偏离 →
  ① 明确承认偏离
  ② 给出偏离的合理性解释（如"数据规模不同""硬件限制"等）
  ③ 评估偏离对最终结论的影响程度
→ 不能默默跳过
```

**三阶段检查：**

| 阶段 | 检查内容 |
|------|---------|
| 事前 | 列出本环节的所有假设 → 逐一检查是否合理 |
| 事中 | 实验结果和假设预期不符时 → 回溯检查假设是否成立 |
| 事后 | 结论部分明确列出"本实验的假设与局限性" |

---

### 1.4 实验设计六维度对齐

**原则：设计实验前，检查六个维度的内在一致性，防止设计缺陷导致实验无效。**

> 基于 Campbell & Stanley (1963) 实验效度理论，扩展为 ML 场景的六维度检查。

| 维度 | 对齐检查 | 典型失效 |
|------|---------|---------|
| 问题→方法 | 问题类型决定了应该用什么方法？ | 分类问题用了回归方法 |
| 数据→任务 | 数据的噪音水平/标注质量匹配任务需求？ | 用干净数据测清洗 pipeline（无效果） |
| 模型→数据 | 模型容量与数据规模匹配？ | 10K 数据训 7B 模型（过拟合） |
| 过程→严谨性 | 训练/验证/测试划分是否有泄漏？ | 验证集和训练集有重叠 |
| 指标→目标 | 评估指标是否真正度量了目标？ | 用 Accuracy 评估不均衡数据（掩盖小类性能） |
| 解释→现实 | 统计显著 ≠ 实际有用？效果量是否有实际意义？ | AUC 从 0.81 提升到 0.82，统计显著但无实际价值 |

**事前检查：** 实验开始前，逐行填写对齐表。任何一行不通过则重新设计。

> **案例**：某项目用已清洗的高质量数据集测试清洗 pipeline，效果仅 0.2%（"数据→任务"不对齐）。换用原始脏数据后效果显著。

---

### 1.5 方法论文档（全局地图）

**原则：第一份文档/第一个 Notebook 专门放置项目全局方法论。**

```
内容清单：
  ① 项目整体架构图（数据流、模型流、评估流）
  ② 所有关键概念定义
  ③ 术语缩写表（英文缩写首次出现标注全称）
  ④ 每个后续模块的五要素摘要
  ⑤ 参考论文和标准值汇总表
  ⑥ 指标口径定义（全项目 Single Source of Truth）
```

**三阶段检查：**

| 阶段 | 检查内容 |
|------|---------|
| 事前 | 方法论文档是否已完成？是否覆盖所有后续模块？ |
| 事中 | 新增模块时 → 回到方法论文档补充五要素摘要 |
| 事后 | 方法论文档与实际执行是否一致？有无遗漏的模块？ |

---

### 1.6 叙事框架

**原则：用结构化叙事组织文档和分析报告。**

**金字塔原理（Barbara Minto, 1985）：**
```
每个分析单元开头第一段：
  结论："本模块证明了 X 方法对 Y 指标的贡献是 Z"
  → 读者 3 秒钟知道核心价值

然后展开支撑（不超过 3 个主要论点）：
  论点 1 + 数据支撑
  论点 2 + 数据支撑
  论点 3 + 数据支撑
```

**SCR 叙事（McKinsey）：**
```
每个实验的"为什么做"：
  S（Situation，情境）：现状是什么
  C（Complication，矛盾）：遇到什么问题
  R（Resolution，解决）：怎么解决的
```

> **案例**：
> S："内容安全审核每天处理 10 亿条数据"
> C："全用大模型成本不可承受，纯规则漏判率高"
> R："级联架构：轻量模型初筛 + 中等模型中筛 + 大模型精筛，成本降 80 倍"

## 二、数据准备阶段（Data Preparation）

### 2.1 数据源与任务匹配

**原则：选择输入数据前先问——"这个数据的噪音水平/难度适不适合当前任务？"** [→ CLAUDE.md]

| 任务类型 | 需要的数据特征 | 反模式 |
|---------|--------------|--------|
| 数据清洗 pipeline | 脏数据（噪音率 >30%） | 用已清洗数据当输入（效果为零） |
| 模型训练 | 干净、有标签的数据 | 用未清洗的原始数据直接训练 |
| 评估/对比 | 有 ground truth 或独立评估器 | 用 pipeline 自身打分评估自身 |

**三阶段检查：**

| 阶段 | 检查内容 |
|------|---------|
| 事前 | 数据噪音率是否在合理范围？是否做过 EDA（探索性分析）？ |
| 事中 | 中间结果是否符合数据特征的预期？如过滤率是否合理？ |
| 事后 | 最终效果是否与数据噪音水平一致？效果过小需检查数据选择 |

---

### 2.2 数据质量多维度画像

**原则：对输入数据做系统性质量评估，使用标准化的质量维度框架。**

> 参考 Wang & Strong (1996) Data Quality Dimensions 框架（被引 6000+次）。

**常用质量维度：**

| 维度 | 英文 | 检查内容 |
|------|------|---------|
| 完整性 | Completeness | 缺失值比例、必填字段覆盖率 |
| 一致性 | Consistency | 同一实体在不同字段/表中是否矛盾 |
| 准确性 | Accuracy | 数据与真实值的偏差程度 |
| 时效性 | Timeliness | 数据是否过期、更新频率是否足够 |
| 适量性 | Appropriate Amount | 数据量是否满足任务最低要求 |
| 可表示性 | Representational Consistency | 格式是否统一、编码是否一致 |

**三阶段检查：**

| 阶段 | 检查内容 |
|------|---------|
| 事前 | 选择与任务相关的质量维度，定义每个维度的达标标准 |
| 事中 | 按维度逐一检查，记录各维度的实际值 |
| 事后 | 生成数据质量报告卡（每个维度一行：维度/标准/实际值/是否达标） |

---

### 2.3 样本量倒推

**原则：确定样本量时，从 pipeline 最末端往前推算，确保每个中间步骤都有足够样本。** [→ CLAUDE.md]

```
推算方法：
  最终需要 X 条有效输出
    ← 最后一步保留率 P_n → 输入需 X / P_n
      ← 前一步保留率 P_(n-1) → 需 X / (P_n × P_(n-1))
        ← ... 推到原始输入量

约束：每个中间步骤的输出不少于 50 条（否则统计无意义）
```

**三阶段检查：**

| 阶段 | 检查内容 |
|------|---------|
| 事前 | 按论文参考保留率倒推输入量 → 确认数据够用 |
| 事中 | 每步执行后检查实际输出量 → 低于 50 条则增大输入 |
| 事后 | 最终样本量是否足够支撑统计结论？ |

> **案例**：某项目初始用 1000 条，经 Step1(40%) → Step2(10%) 后仅剩 40 条，无统计意义。改为倒推后确定 12000 条输入。

---

### 2.4 训练数据分布对齐（Covariate Shift 控制）

**原则：训练数据与推理数据在干扰变量（Nuisance Variables）上应保持同分布，避免模型学习虚假相关（Spurious Correlation）。** [→ CLAUDE.md]

> 理论基础：Covariate Shift（协变量偏移，Shimodaira 2000）。当训练和推理数据在非目标特征维度（如文本长度、语言分布等）上存在系统性差异时，模型可能利用这些差异作为捷径（Shortcut Learning, Geirhos et al. 2020），导致泛化性能下降。

**关键实践：**
- 正样本（参考标准）保持完整，不做截断
- 负样本若在干扰维度上与正样本差异显著（如长度比 >2x），需做归一化对齐
- 推理时保持与训练时相同的预处理（train-inference 一致性）

**三阶段检查：**

| 阶段 | 检查内容 |
|------|---------|
| 事前 | 对比正/负样本在各维度的分布 → 差异 >2x 的维度需要对齐 |
| 事中 | 训练后检查模型是否利用了干扰变量（如：长文档一律高分？） |
| 事后 | 推理时的预处理是否与训练时一致？ |

> **案例**：某分类器的正样本平均 200 词，负样本平均 800 词。未对齐时，模型学到"短=好"的虚假相关，分数全部压缩在 [0.50, 0.55]。对齐文本长度后，分离度从 <0.1 提升到 >0.8。

---

### 2.5 数据血缘追踪（自动化）

**原则：Agent 通过 AST 静态分析自动构建脚本间的数据依赖图，无需人工登记。** [→ CLAUDE.md]

**三阶段全由 Agent 自动完成：**

| 阶段 | Agent 操作 | 方法 |
|------|-----------|------|
| 事前 | 扫描所有脚本，解析文件读写调用，生成依赖表 `{脚本: [输入], [输出]}` | AST 解析 `open()`/`read_*()`/`Path()` 等调用 |
| 事中 | 脚本修改后，自动识别受影响的下游脚本并提示 | 增量扫描：修改文件的输出 → 反查下游消费者 |
| 事后 | Pipeline 执行完，校验所有输出文件存在且非空，时间戳晚于输入 | `Path.exists()` + `stat()` 检查 |

**依赖表格式示例：**
```
| 脚本 | 输入文件 | 输出文件 | 下游消费者 |
|------|---------|---------|-----------|
| run_step1.py | raw_data.jsonl | step1_output.jsonl, step1_stats.json | run_step2.py, NB02 |
| run_step2.py | step1_output.jsonl | step2_output.jsonl | run_step3.py, NB03 |
```

**影响分析：** 修改 `run_step1.py` 时，Agent 自动提示 "run_step2.py 和 NB02 依赖 step1_output.jsonl，可能需要重新执行"。

## 三、实现与执行阶段（Implement & Execute）

### 3.1 Pipeline 节点标准化

**原则：串行 Pipeline 中的每一个节点都有标准定义卡片。**

**节点卡片模板：**

```
| 项目 | 内容 |
|------|------|
| 节点名称 | （如：文本安全分类器训练） |
| 输入 | 数据文件 + 格式 + 预期数量 |
| 输出 | 产出文件 + 格式 + 预期数量 |
| 处理逻辑 | 简要描述核心算法 |
| 基准值 | 论文/业界的标准结果 |
| 预期值 | 我们期望达到的值（可以低于基准，但要说明原因） |
| 偏离说明 | 我们的条件与论文的差异及其影响 |
| 验收检查 | 怎么确认这一步做对了 |
| 观察重点 | 最可能出问题的地方 |
| 失败处理 | 如果未达标，下一步怎么办 |
```

**三阶段检查：**

| 阶段 | 检查内容 |
|------|---------|
| 事前 | 所有节点卡片是否填写完整？基准值是否有来源？ |
| 事中 | 执行结果是否在预期值范围内？偏差过大则暂停排查 |
| 事后 | 所有节点的验收检查是否通过？观察重点是否有异常？ |

---

### 3.2 Pipeline 漏斗指标体系

**原则：多步 Pipeline 必须有按执行顺序的漏斗表，从大指标细拆到子指标。** [→ CLAUDE.md]

**漏斗表模板：**

```
| 阶段 | 子步骤 | 输入 | 输出 | 条件过滤率 | 累计保留率 | 参考值 | 差异分析 |
|------|-------|------|------|-----------|-----------|--------|---------|
| Step1 整体 | — | 12000 | 4800 | 60.0% | 40.0% | 40-60% | 正常 |
| └ 子步骤A | 规则A | 12000 | 8400 | 30.0% | 70.0% | 25-35% | 正常 |
| └ 子步骤B | 规则B | 8400 | 4800 | 42.9% | 40.0% | 35-45% | 正常 |
```

**关键约束：**
- 必须从大步骤细拆到子步骤（如 quality_filter → 各子规则）
- 条件过滤率 = 该步丢弃数 / 该步输入数（不是原始总量）
- 累计保留率 = 该步输出 / 原始输入

---

### 3.3 配置驱动规模

**原则：所有可变参数集中在配置文件中，代码不硬编码数值。** [→ CLAUDE.md]

```
核心做法：
  ① 集中式配置文件（如 YAML/JSON）定义所有可变参数
  ② 代码通过配置读取器获取参数值
  ③ 多档运行通过切换配置档位实现（如 smoke_test / full_run）
  ④ 配置值沿调用链逐层传递到最终执行组件

参考实现：
  configs/run_config.yaml 定义 doc_limit, batch_size, epochs 等
  代码中 config["doc_limit"] 读取，不写死 3000 或 100000
```

---

### 3.4 增量执行验证

**原则：写一步、跑一步、验一步。步骤 N 未通过验收，禁止开始步骤 N+1。** [→ CLAUDE.md]

**验收标准速查：**

| 产物类型 | 验收标准 |
|---------|---------|
| 脚本 | 退出码 0 + 预期输出文件存在且非空 |
| 分析报告 | 所有代码单元有输出，无报错 |
| 配置文件 | 可加载 + 关键字段有值 |
| 模型文件 | 文件存在 + 可加载 + sanity check 通过 |

**报告完成时必须附带证据：**
- 日志摘要（关键输出行）
- 产出物清单（文件名 + 大小）
- 核心指标（数量/比率/得分）

> **案例**：某项目 Agent 报告"5 个阶段全部完成"，但实际上 5 个脚本从未被执行。加入增量验证后，每步执行后立即检查输出，杜绝"代码存在 ≠ 任务完成"。

---

### 3.5 多工作点设计（Operating Point Selection）

**原则：同一评分模型在不同使用场景下选择不同决策阈值。**

> 理论基础：ROC 分析中的工作点选择（Fawcett 2006, "An Introduction to ROC Analysis"）。

```
场景示例：
  运行时路由（宽松阈值）→ 优化召回，减少漏判
  分析评估（严格阈值）→ 优化精确度，量化效果

原因：不同场景的误判成本不同。
  运行时漏判一条坏内容 → 用户投诉（成本高）
  分析时多保留几条边界样本 → 影响不大（成本低）
```

**三阶段检查：**

| 阶段 | 检查内容 |
|------|---------|
| 事前 | 是否需要多个工作点？各场景的成本/收益权衡是什么？ |
| 事中 | 阈值切换逻辑是否正确实现？ |
| 事后 | 各工作点的实际精确率/召回率是否符合预期？ |

## 四、分析与评估阶段（Analyze & Evaluate）

### 4.1 输出必有结论

**原则：每个图表、数字、表格产出后，必须紧跟一段分析结论。禁止"为展示而展示"。** [→ CLAUDE.md]

**结论的最低要求（三要素）：**
1. **判断**：该指标是正常、异常、还是需要关注
2. **对比**：与预期基准（论文值/上一版/Baseline）的差距
3. **影响**：对下一步或整体结论的影响

**示例：**

```
不合格：
  [图表：过滤率柱状图]
  （没有任何文字说明）

合格：
  [图表：过滤率柱状图]
  结论：Step1 过滤率 58%，在论文参考范围 40-60% 内，属正常。
  其中子步骤 B 的过滤率 42.9% 略高于参考上限 40%，
  原因是输入数据的噪音水平（~50%）高于论文数据集（~30%）。
  此偏差不影响整体结论，但需在结论中注明。
```

---

### 4.2 层次化评估框架（Hierarchical Evaluation Framework）

**原则：分析结论从原子指标自底向上逐层聚合，形成完整的证据链。** [→ CLAUDE.md]

> 理论基础：多层次分析（Gelman & Hill 2006），证据综合（Cochrane Handbook）。

```
Layer 1：原子指标（Atomic Metrics）
  每个图表/数字产出一个事实性结论
  如："Step1 过滤率 58%，在论文参考范围 40-60% 内"

Layer 2：组件发现（Component Findings）
  将同一组件的多个原子指标综合
  如："Step1 整体健康——过滤率正常、质量提升显著、无异常过滤"

  指标的角色分类（决定如何参与上层结论）：
    ▸ 支撑性指标：直接证明组件有效（如 AUC 提升）
    ▸ 非负向指标：不拖后腿即可（如过度过滤率 <5%）
    ▸ 诊断性指标：异常时才有意义，正常时不参与结论（如数据版本哈希）

Layer 3：系统结论（System Conclusions）
  跨组件对比和综合
  如："三代方法中，第三代在质量-成本权衡上最优"

Layer 4：方法论结论（Methodology Conclusions）
  回答"这个项目验证了什么方法论命题"
  如："级联架构确实能在保持质量的前提下大幅降低成本"
```

**关键要求：每一层的结论都标注其证据来源——由哪些下层指标支撑。**

---

### 4.3 统计差异调查方法论

**原则：发现统计差异较大时（如指标与论文参考值偏差超过 2x），禁止直接下结论，必须先做两步验证。** [→ CLAUDE.md]

```
步骤 1：抽查案例
  抽取 10-100 条样本 → 人工或 LLM 评估 → 判断数据特征

步骤 2：技术核验
  运行定量分析脚本 → 分布统计、子集拆分、条件过滤等

步骤 3：给出结论
  基于步骤 1+2 的证据 → 而非仅凭文字推理
```

> **案例**：某指标 P50=0.09 vs 论文参考值 0.7（偏差 8 倍）。若直接下结论"数据质量极差"则是错误的。技术核验发现 72.7% 的行是导航短行（≤5 词无标点），仅计算内容行时 P50=0.588，接近论文预期。根因是数据格式特征，而非质量问题。

---

### 4.4 ML 模型健康检查

**原则：ML 模型训练后，必须进行独立于模型的健康检查（Sanity Check），确认模型具备基本区分能力。** [→ CLAUDE.md]

**通用检查框架（阈值需根据任务校准）：**

| 检查项 | 检查内容 | 关注信号 |
|--------|---------|---------|
| 分数分布 | 观察 min/max/mean/std | std 过小 → 模型缺乏区分能力 |
| 正负分离度 | mean(正样本分数) - mean(负样本分数) | 接近 0 → 模型未学到有效信号 |
| 黄金样本 | 已知好样本分数应高于已知坏样本 | 违反 → 模型存在严重问题 |
| 基线对比 | 与随机预测 / 多数类预测对比 | 无显著优势 → 模型无效 |
| 数据版本 | 训练数据文件哈希与预期一致 | 不一致 → 结果不可信 |

**注意：** 上述"关注信号"不设绝对阈值。具体的告警阈值应根据任务类型、数据规模和业务要求在项目初期校准确定。

**三阶段检查：**

| 阶段 | 检查内容 |
|------|---------|
| 事前（训练前） | 正/负样本分布对齐？数据版本正确？标签有无错误？ |
| 事中（训练后） | 五项健康检查全部通过？ |
| 事后（推理时） | 推理结果分布与训练时一致？预处理是否保持 train-inference 一致？ |

> **案例**：某分类器训练后分数全部压缩在 [0.50, 0.55]（std ≈ 0.01），完全无区分能力。排查发现训练数据的负样本来自旧数据集，与当前推理数据分布严重不匹配。重训后分离度从 <0.1 提升到 >0.8。

---

### 4.5 黄金样本验证（Canary Samples）

**原则：选取已知预期结果的样本，在每次 pipeline 运行时验证这些样本的实际结果是否符合预期。** [→ CLAUDE.md]

```
黄金样本集构成（建议 10 条）：
  3 条已知高质量 → 预期通过所有步骤
  3 条已知低质量 → 预期在早期步骤被过滤
  4 条边界样本   → 预期在特定步骤被处理

每条样本标注：
  expected_outcome: pass / fail / boundary
  expected_filter_stage: 预期在哪个阶段被过滤
  selection_reason: 为什么选这条

三层验证：
  ① 每步 pipeline 后检查黄金样本的去留是否符合预期
  ② 代码变更后快速回归验证（30 秒内完成）
  ③ 违反预期 → 立即停下排查
```

---

### 4.6 独立外部评估（Independent External Evaluation）

**原则：使用与 pipeline 无关的独立评估器验证输出质量。** [→ CLAUDE.md]

> 术语来源：Extrinsic Evaluation（NLP 传统），Independent External Validation（医学 ML）。

```
独立性要求：
  ① 训练数据与 pipeline 不重叠
  ② 模型架构或超参数与 pipeline 内组件不同
  ③ 评估器不参与 pipeline 的任何决策

常见形式：
  独立训练的质量分类器
  语言模型困惑度（Perplexity）
  人工抽样评审（小规模但高精度）
  下游任务性能（如在标准基准上的表现）
```

---

### 4.7 实验记录与可复现性

**原则：每个实验必须记录完整元信息，确保可复现。**

> 参考：Joelle Pineau (2019) "ML Reproducibility Checklist"; CRISP-ML(Q) (Studer et al. 2021)。

**实验记录表模板：**

```
| 实验名 | 假设 | 变量设置 | 指标1 | 指标2 | 结论 | 意外发现 |
|--------|------|---------|-------|-------|------|---------|
| Baseline | — | 默认参数 | 0.82 | 91% | 基准 | — |
| +方法A | A能提升X | 改变参数P | 0.85 | 94% | 假设成立 | — |
| -模块B | B是否必要 | 移除模块B | 0.80 | 95% | B 必要 | 移除后指标2 反升 |
```

**可复现性清单：**

```
☐ 代码版本（Git commit hash）
☐ 数据版本（文件哈希或版本号）
☐ 随机种子（固定且报告）
☐ 超参数完整列表
☐ 运行环境（语言版本、硬件、关键依赖版本）
☐ 训练时长
☐ 不同随机种子下结果是否稳定（至少 2 次）
```

## 五、总结与交付阶段（Summarize & Deliver）

### 5.1 论文标准值口径核验

**原则：引用论文方法或结论时，必须做口径核验——我的实验条件和论文是否一致。**

```
核验表模板：
| 对比项 | 论文条件 | 我的条件 | 影响评估 |
|--------|---------|---------|---------|
| 数据量 | 240T token | 50K 文档 | 结论趋势一致但绝对值会低 |
| 模型 | 7B 参数 | 125M 参数 | 可能有能力涌现差异 |
| 训练时长 | 完整训练 | 5 epoch | 效果未充分收敛 |
```

**规则：**
- 方法论文档（事前）：只放论文参考值，不放实际值
- 分析报告（事后）：论文参考值 + 本项目实际值，对比分析

---

### 5.2 统一指标与术语

**原则：所有指标的口径在方法论文档中统一定义，全项目引用同一定义（Single Source of Truth）。** [→ CLAUDE.md]

```
规则：
  ① 新指标出现 → 先在方法论文档中定义口径（分子/分母/公式）
  ② 英文缩写 → 全文第一次出现标注全称
  ③ 同一指标在不同模块中计算口径必须一致
  ④ 后续引用只引用，不重复定义
  ⑤ 百分比/比率必须同时说明分子和分母
  ⑥ 区分绝对差（+10 个百分点）和相对变化率（增长了 10%）
```

---

### 5.3 文档结构规范

**原则：技术文档遵循标准结构，确保可读性和完整性。**

**文档开头三问（前 50 行必答）：**
1. **问题**：现状与痛点（1-2 句）
2. **方案**：核心做法（1-2 句）
3. **预期**：关键指标的预期值（量化）

**ML 报告标准结构**（参考 UT Austin CS 391L）：
```
1. Introduction：问题 + 动机 + 方法预览 + 结论预览
2. Problem & Algorithm：输入输出定义 + 算法描述
3. Experimental Evaluation：
   3.1 方法论（假设/变量/数据/对比方法）
   3.2 结果（数字和图表）
   3.3 讨论（结果说明什么？为什么？意外发现？）
4. Related Work：与已有方法的异同
5. Conclusion：主要发现 + 局限性 + 未来方向
```

**Diataxis 分层**——每节只写一种类型的内容：

| 类型 | 核心动词 | 内容 |
|------|---------|------|
| Explanation（为什么） | 解释、对比 | 背景、选型理由 |
| Reference（是什么） | 定义、列表 | 指标口径、术语表 |
| How-to（怎么做） | 执行、验证 | 步骤列表、核对清单 |

## 六、贯穿原则

### 6.1 Agent 指令消歧与自检

**原则：Agent 收到模糊指令时主动消歧；每步执行后自检输出。** [→ CLAUDE.md]

**指令消歧（Disambiguation）：**
```
当用户指令可能有多种理解时：
  → Agent 主动列出可能的解释，请用户选择

示例：
  用户："优化一下数据质量"
  Agent："这个指令有多种理解，请确认：
    A) 调整过滤阈值提高保留率
    B) 重训分类器提升质量分数
    C) 其他（请说明）"
```

**执行自检：**
```
每步执行后，Agent 自检：
  ① 输出文件是否存在且非空？
  ② 核心指标是否在合理范围？
  ③ 是否与五要素中的"预期"一致？
  ④ 是否遗漏了用户的任何子需求？（多任务核销）
```

---

### 6.2 跨组件一致性检查

**原则：跨脚本、跨模块的指标定义、数据格式、术语使用必须一致。** [→ CLAUDE.md]

```
检查方法：
  ① 每个模块完成后，对照方法论文档检查术语和指标口径
  ② 同一指标在不同模块中的计算结果不应有不合理的差异
  ③ 数据格式在上下游之间保持一致（字段名、编码、分隔符等）
```

---

### 6.3 规则管理架构

**原则：规则按通用性分层管理，新项目可高效复用，新规则可自动化纳入。**

```
三层体系：

Layer 1：全局规则（~/.claude/CLAUDE.md）
  ├── 适用：所有项目自动生效
  ├── 内容：执行纪律、架构契约、文档规范、变更保护
  └── 新项目零配置继承

Layer 2：项目规则（项目/MEMORY.md）
  ├── 适用：仅当前项目
  ├── 内容：项目特有状态、路径、决策、教训
  └── 项目间互不干扰

Layer 3：方法论文档（docs/STANDARD_METHODOLOGY.md）
  ├── 适用：同类项目的详细参考
  ├── 内容：完整方法论 + 案例 + 参考文献
  └── 新项目通过引用激活

新规则纳入流程（Agent 自动执行）：

  Step 1 识别：Agent 在执行中发现潜在新规则时（如踩坑、发现反模式、用户纠正等），
         主动向用户提出："发现一条潜在方法论规则：[规则描述]，是否纳入？"

  Step 2 确认：用户确认后，Agent 判断规则类型并写入对应位置：
         ① 全局通用 → 写入 ~/.claude/CLAUDE.md（Layer 1）
         ② 项目特有 → 写入 MEMORY.md（Layer 2）
         ③ 方法论性质 → 写入 docs/METHODOLOGY_DELTA.md（增量文件），不直接修改本文档
         ④ 若需 Agent 自动执行 → 必须同时写入 Layer 1（CLAUDE.md）

  Step 3 提醒合并：当 METHODOLOGY_DELTA.md 累积达到 5 条以上已验证规则时，
         Agent 主动提醒用户："DELTA 中已有 N 条已验证规则待合并，是否执行合并流程？"

新项目启动：
  ① Layer 1 自动继承
  ② 复制 STANDARD_METHODOLOGY.md + METHODOLOGY_COMPACT.md 到新项目 docs/
  ③ 用 METHODOLOGY_COMPACT.md 覆盖项目 CLAUDE.md
  ④ Layer 2 从模板初始化
```

**增量规则文件（docs/METHODOLOGY_DELTA.md）：**

每个项目独立维护一份"新规则收集箱"，不直接修改 `STANDARD_METHODOLOGY.md`：

```markdown
# 方法论增量规则

## 来自项目 X

### 新规则 1：[标题]
- 发现日期：YYYY-MM-DD
- 触发场景：什么情况下暴露了这个问题
- 规则内容：具体的方法论规则
- 验证状态：已验证 / 待验证

### 新规则 2：...
```

**定期合并流程（项目结束或阶段性回顾时）：**

```
合并步骤：
  ① 读取所有项目的 docs/METHODOLOGY_DELTA.md
  ② 筛选"已验证"的规则
  ③ 合并到 docs/STANDARD_METHODOLOGY.md 对应章节
  ④ 重新复制到其他项目
  ⑤ 清空已合并的 DELTA 条目
```

**为什么不直接改主文档：**
- **审计追溯**：每条新规则记录了发现日期、来源项目、触发场景
- **检查门禁**：只有"已验证"的规则才会合并到主文档
- **主文档稳定**：STANDARD_METHODOLOGY.md 始终是经过验证的稳定版本
